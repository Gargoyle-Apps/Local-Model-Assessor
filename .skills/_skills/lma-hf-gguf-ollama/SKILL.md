# LMA HF GGUF → Ollama

## When to use this skill

Load when the user has a **GGUF file** (or HF model card) for a model that is **not** in the Ollama library and wants to import it into the LMA assessment workflow.

## Instructions

### 1. Pre-flight: hardware check

Read `computer-profile/hardware-profile.yaml` and confirm the GGUF size / quant fits the machine's VRAM budget. Do not proceed if it won't fit.

### 2. Understand the key semantics

| Field | Meaning |
|-------|---------|
| `models.model_id` | Ollama name for the **base** weights (e.g. `myrepo-model:q4_k_m`). Must match what appears in `ollama list` after import. |
| `models.url` | HF model card or direct file link (provenance / human reference). |
| `models.install` | Command to get the base into Ollama. For GGUF: `ollama create <model_id> -f <path-to-base-modelfile>`. |
| `provisioned_models.pull_command` | Mirrors `models.install` — for GGUF bases this is `ollama create -f …`, not `ollama pull`. Treat as "base install step." |

### 3. Create the base Modelfile

This is **not** generated by `add-model-from-yaml.py` — create it manually:

```
model-data/modelfile/<model-name>--<quant>-base.mf
```

Minimum content:

```
FROM /absolute/path/to/model.gguf
```

**Path resolution:** Ollama resolves `FROM` paths relative to the **working directory where you run `ollama create`**, not relative to the Modelfile's location. Use an absolute path or ensure the CWD is correct.

If the model needs a specific chat template, add `TEMPLATE` to this file. Document download location and any template needs in `model_docs.caveats`.

### 4. Choose a stable `model_id`

Use a name that will be stable in `ollama list`. Set `install` to the exact `ollama create … -f …` command, using a path from the repo root if the GGUF is repo-relative.

### 5. Fill the assessment YAML

Follow the standard assessment flow from `lma-assess-import-model`:
1. Assess using `LLM-prompts/model-assessment-prompt.yaml`.
2. Write output to `model-data/new-models.yaml`.
3. Include `provisioning` aligned with `by_role` (same rules as catalog models).

### 6. Import and export

```bash
./scripts/py scripts/add-model-from-yaml.py [--assessor NAME --assessor-type TYPE] model-data/new-models.yaml
./scripts/py scripts/export-assessed-models.py
```

### 7. Verify

1. User runs the base import: the `install` command from the YAML (e.g. `ollama create myrepo-model:q4_k_m -f model-data/modelfile/myrepo-model--q4_k_m-base.mf`).
2. Confirm with `ollama list`.
3. Run provisioned role clone `create_command` values if needed.

### Checklist

- [ ] Read `hardware-profile.yaml`; confirm GGUF size / quant fits.
- [ ] User has (or will) download the `.gguf` to a known path; base `.mf` references that path.
- [ ] Stable `model_id` chosen; `install` set to exact `ollama create … -f …`.
- [ ] Assessment YAML filled including `provisioning` aligned with `by_role`.
- [ ] Ran `add-model-from-yaml.py` and `export-assessed-models.py`.
- [ ] Verified with `ollama list` after base import; ran clone `create_command` if needed.

## Notes

- Base `.mf` files go under `model-data/modelfile/` (gitignored except `.gitkeep`).
- Role clone `.mf` files are generated by `add-model-from-yaml.py` with `FROM <base model_id>` + parameters.
- See [Ollama import docs](https://docs.ollama.com/import) and [Modelfile reference](https://docs.ollama.com/modelfile) for GGUF-specific options.
